# Building Something New Workflow

You have an idea for something you want to create or add to your project, and you want Codex to help you build it. This workflow guides you through describing what you want, reviewing the plan, and working with Codex to bring it to life, step by step.

---

## When to Use This Workflow

- You want to add something new to an existing project
- You want to create a new file, document, or set of files from scratch
- You have an idea but are not sure how to implement it
- You want to reorganize or restructure something

---

## Step 1: Describe What You Want

Start by telling Codex what you want to build. You do not need to know the technical details. Focus on what the end result should look like and what it should do.

**Use this simple template:**

> "I want to build [what it is]. It should [what it does]. The end result should look like [description of the finished product]."

### Examples

**Simple:**
> "I want to create a contact list that keeps track of names, email addresses, and phone numbers. It should be easy to search and sorted alphabetically."

**More detailed:**
> "I want to add a monthly summary section to my project. It should automatically list all the files that were changed in the last month, group them by folder, and show the date each one was last modified. The summary should be saved as a text file called 'monthly-changes.txt'."

**Analogy:** Think of it like describing a room you want remodeled. You might say "I want a home office with built-in bookshelves, good lighting, and room for two monitors." You do not need to specify the type of wood or the exact electrical wiring. You describe what you want, and the builder figures out the details.

---

## Step 2: Ask for a Plan Before Building

For anything beyond a tiny change, ask Codex to propose a plan before it starts making changes. This is the Plan-Validate-Execute approach.

**What to say:**

> "Before you start building, outline your plan. Tell me:
> 1. What files you will create or change
> 2. What approach you will take
> 3. What the result will look like
> 4. Are there any decisions you need my input on?"

Codex will respond with a structured plan. Read through it carefully.

### What to Look For in the Plan

- **Does it match your vision?** Make sure Codex understood what you wanted.
- **Is the scope right?** Is it doing too much or too little?
- **Are there surprises?** Anything you did not expect?
- **Decision points:** Has Codex identified any choices that need your input?

---

## Step 3: Refine the Plan

It is rare for the first plan to be exactly right. Go back and forth with Codex until you are happy.

**What to say:**

> "I like the overall approach, but I want to change a few things:
> - [thing you want changed]
> - [thing you want added]
> - [thing you want removed]"

Or ask questions:

> "Why did you choose to do it that way? Is there a simpler approach?"

> "What if I wanted it to also include [additional feature]?"

**Analogy:** This is like reviewing architectural drawings before construction starts. It is much easier (and cheaper) to move a wall on paper than after it is built.

---

## Step 4: Build Step by Step

Once you are happy with the plan, ask Codex to build it incrementally rather than all at once. This lets you review each piece as it is created.

**What to say:**

> "The plan looks good. Start building it, but do it one step at a time. After each step, show me what you did and wait for my approval before continuing."

### Why Step by Step?

- You can catch problems early before they cascade
- You understand what is being built as it happens
- You can change direction mid-stream without wasting work
- It is easier to review small changes than one massive change

---

## Step 5: Review Each Change

After each step, Codex will show you what it created or changed. Take a moment to review.

**Questions to ask yourself:**
- Does this look right?
- Does it match what I asked for?
- Is there anything unexpected?

**Questions to ask Codex:**
> "Walk me through what you just did in simple terms."

> "Is this consistent with what we planned?"

> "What is the next step?"

If something is not right:

> "That is not quite what I wanted. Can you change [specific thing]?"

---

## Step 6: Verify the Final Result

Once all the steps are complete, ask Codex to verify that everything works together.

**What to say:**

> "Everything looks done. Can you review the final result and make sure:
> 1. It does what we planned
> 2. Everything works together properly
> 3. There are no issues or leftover problems"

Codex will do a final check and report back.

---

## Step 7: Document What Was Built

If this is something others will use or something you will come back to later, ask Codex to write a short description.

**What to say:**

> "Can you write a short description of what we built, how it works, and how to use it? Keep it simple enough for someone who was not part of this conversation."

This saves future-you (or your colleagues) from having to figure it out from scratch.

---

## Tips for Building Successfully

### Start Small, Then Expand
If your vision is big, start with the simplest possible version first. Get that working, then add features one at a time. This is almost always faster and more reliable than trying to build everything at once.

> "Let's start with a basic version that just does [core feature]. We can add the extra features later."

### Be Specific About What "Done" Looks Like
The clearest way to communicate your vision is to describe the finished product:

> "When this is done, I should be able to [specific thing], and the result should look like [specific description]."

### Use Examples
If you can show Codex an example of something similar to what you want, that is extremely helpful:

> "I want something similar to [this existing file], but with [these changes]."

### Save Your Progress
If you are building something large and your context is getting full, save the plan and current progress, then start a fresh session:

> "Summarize everything we have built so far and what still remains. I will use this summary to continue in a new session."

### Do Not Be Afraid to Start Over
If things have gone off track and the result is not what you wanted, it is often faster to start a new session with clearer instructions than to try to fix a mess.

## Example: Building Something New From Start to Finish

**You:** "I want to create a weekly status report template. It should have sections for what was accomplished this week, what is planned for next week, any blockers, and a section for notes. I want it as a simple text file I can fill in each week."

**Codex:** "Here is my plan: I will create a file called 'weekly-status-template.txt' with four clearly labeled sections: Accomplishments, Next Week Plans, Blockers, and Notes. Each section will have brief instructions and placeholder text showing what to fill in. Shall I proceed?"

**You:** "That sounds good, but can you also add a date field at the top and a section for hours worked?"

**Codex:** "Updated plan: The template will start with a Date field and your Name field at the top, followed by five sections: Hours Worked, Accomplishments, Next Week Plans, Blockers, and Notes. Each with instructions and placeholders. Ready to build?"

**You:** "Go for it."

**Codex:** [creates the file and shows the contents]

**You:** "That looks great. Can you also create an example filled-in version so I can see what a completed report looks like?"

**Codex:** [creates an example file with realistic sample data]

**You:** "Perfect. Can you write a quick note about how to use these templates?"

**Codex:** "Here is a short usage guide: 'Copy the template file at the start of each week. Fill in the date and your name. Complete each section with that week's information. Save the file with the week's date in the filename (e.g., status-2026-03-28.txt).'"

And just like that, you have a complete, usable set of files built from a simple idea.
